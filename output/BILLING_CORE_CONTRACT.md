# 1. Billing Doctrine

Billing is the authoritative financial-state engine of the ISP infrastructure.

Billing is responsible for determining financial truth.

Billing answers:

* who owes money
* what is owed
* why it is owed
* when it becomes due
* when it becomes overdue
* whether a subscriber remains financially eligible for service
* whether downstream enforcement becomes legally valid

Billing does not execute payment collection.

Billing does not suspend service.

Billing does not restore service.

Billing does not perform network enforcement.

Billing does not mutate Radius policy directly.

Billing does not communicate with MikroTik directly.

Billing only produces deterministic financial state.

## 1.1 Financial Truth Doctrine

Financial truth must be:

* deterministic
* replay-safe
* auditable
* immutable where required
* recoverable from authoritative events
* free from hidden manual mutation
* independent from network runtime state

No system other than Billing Core may create financial debt.

No system other than Billing Core may define invoice validity.

No system other than Billing Core may determine financial eligibility.

# 3. Billing Entity Model

## 3.1 Subscriber

A subscriber is the billable customer subject.

Billing Core does not own subscriber identity creation.

Billing Core references the subscriber as a billing subject.

A subscriber may own one or more billable services.

## 3.2 Service

A service is a billable subscription unit.

A service may generate recurring charges.

A service may become financially eligible or ineligible.

Billing Core determines financial eligibility.

Billing Core does not provision or deprovision the service.

## 3.3 Invoice

An invoice is an immutable financial claim generated by Billing Core.

An invoice contains one or more charges.

An invoice may reference credits or adjustments.

An invoice becomes debt when it reaches maturity and remains unpaid.

## 3.4 Charge

A charge is a financial line item produced by deterministic billing rules.

Charge types include:

* recurring charge
* one-time charge
* prorated charge
* adjustment charge

A charge must have a traceable generation cause.

## 3.5 Debt

Debt is an outstanding unpaid financial obligation derived from valid invoices.

Debt is not manually created.

Debt must always be traceable to invoice authority.

## 3.6 Credit

Credit is a positive financial balance applied against subscriber obligation.

Credit must be traceable to payment settlement, correction, or authorized adjustment.

## 3.7 Adjustment

An adjustment is an authorized correction to financial state.

Adjustments must never mutate historical invoices directly.

Adjustments must create new auditable financial events.

# 4. Billing Lifecycle Model

Billing lifecycle transitions must be deterministic.

## 4.1 Service Creation

Service creation alone does not create debt.

A service becomes billable only after Billing Core receives valid activation authority.

## 4.2 Billing Activation

Billing activation defines when financial responsibility begins.

Billing activation must be explicit, auditable, and replay-safe.

## 4.3 Invoice Generation

Invoices must be generated only by Billing Core.

Invoice generation must be idempotent.

The same billing period must not produce duplicate valid invoices for the same service unless explicitly corrected through an adjustment event.

## 4.4 Invoice Maturity

An invoice becomes due according to its deterministic due date.

Due date calculation must be reproducible from billing rules and source events.

## 4.5 Overdue State

An invoice becomes overdue only after its due boundary has passed and unpaid debt remains.

Overdue status must not depend on UI state, reminder state, manual notes, or enforcement state.

## 4.6 Service Termination

Service termination stops future billing generation.

Service termination does not erase historical debt.

Historical debt remains valid until resolved, adjusted, or legally closed.

# 5. Invoice Authority Model

## 5.1 Invoice Creation

Only Billing Core may create invoices.

Invoices must be generated from:

* subscriber reference
* service reference
* billing period
* billing rule
* charge source
* generation timestamp or billing-cycle marker
* deterministic invoice identity

## 5.2 Invoice Immutability

A finalized invoice must not be manually mutated.

Corrections must be represented through:

* credit note
* debit adjustment
* correction invoice
* reversal event

Historical invoice truth must remain preserved.

## 5.3 Invoice Validity

An invoice is valid only if:

* generated by Billing Core
* linked to an authorized service
* linked to a valid billing period
* contains deterministic charges
* has no duplicate authoritative invoice for the same billing scope
* passes reconciliation validation

## 5.4 Invoice Lifecycle

Allowed invoice states:

* DRAFT
* ISSUED
* DUE
* PARTIALLY_PAID
* PAID
* OVERDUE
* DISPUTED
* VOIDED
* CLOSED

Forbidden invoice behavior:

* paid invoice mutation
* silent deletion
* manual amount overwrite
* duplicate invoice replacement without reversal
* payment system creating invoice truth

# 6. Charge Generation Model

## 6.1 Recurring Charges

Recurring charges are generated from active billable services.

Recurring charge generation must be:

* period-bound
* subscriber-bound
* service-bound
* idempotent
* replay-safe

## 6.2 One-Time Charges

One-time charges require explicit authorized cause.

Examples:

* installation fee
* device fee
* migration fee
* penalty fee
* manual approved service charge

One-time charges must not be hidden debt.

## 6.3 Adjustments

Adjustments must be explicit financial events.

Adjustments may reduce or increase balance.

Adjustments must never rewrite previous financial history.

## 6.4 Replay-Safe Charge Law

For the same subscriber, service, charge rule, and billing period, Billing Core must produce the same charge result during replay.

Duplicate charge generation is invalid unless caused by a distinct authorized financial event.

# 7. Debt State Model

Debt is derived from unpaid valid invoices.

Allowed debt states:

* CURRENT
* DUE
* OVERDUE
* DISPUTED
* RESOLVED

## 7.1 CURRENT

Debt exists but has not yet reached due boundary.

## 7.2 DUE

Debt has reached due boundary but has not exceeded overdue boundary.

## 7.3 OVERDUE

Debt remains unpaid after due boundary.

## 7.4 DISPUTED

Debt is under formal review.

Dispute does not delete debt.

Dispute may temporarily change eligibility rules depending on policy.

## 7.5 RESOLVED

Debt is resolved through:

* full payment
* credit application
* authorized adjustment
* write-off
* legal closure

Resolved debt must remain auditable.

# 8. Service Eligibility Model

Billing Core may determine financial eligibility.

Billing Core may not enforce eligibility.

## 8.1 Eligibility Classes

Billing Core may classify service eligibility as:

* ELIGIBLE
* PAYMENT_DUE
* OVERDUE
* DELINQUENT
* TERMINATION_PENDING
* FINANCIALLY_CLOSED

## 8.2 Subscriber Service Conditions

| Subscriber Condition  | Billing Meaning                       | Enforcement Authority |
| --------------------- | ------------------------------------- | --------------------- |
| Active subscriber     | Financially valid                     | Not Billing           |
| Due subscriber        | Payment required                      | Not Billing           |
| Overdue subscriber    | Enforcement may become valid          | Not Billing           |
| Suspended subscriber  | Enforcement already applied elsewhere | Not Billing           |
| Terminated subscriber | Billing stops future cycles           | Not Billing           |

Billing produces eligibility state.

Policy Engine translates eligibility into policy intent.

FreeRADIUS executes policy.

MikroTik enforces policy.

# 9. Billing State Machine

Billing Core state machine:

* PENDING
* ACTIVE
* DUE
* OVERDUE
* DELINQUENT
* CLOSED

## 9.1 State Definitions

### PENDING

Billing subject exists but billing is not yet active.

### ACTIVE

Billing is active and no due debt exists.

### DUE

A valid invoice exists and payment is required.

### OVERDUE

Due invoice remains unpaid after due boundary.

### DELINQUENT

Overdue debt has crossed the delinquency boundary.

### CLOSED

Billing lifecycle has ended.

Closed state does not erase historical debt.

## 9.2 Legal Transitions

| From       | To         |
| ---------- | ---------- |
| PENDING    | ACTIVE     |
| ACTIVE     | DUE        |
| DUE        | ACTIVE     |
| DUE        | OVERDUE    |
| OVERDUE    | ACTIVE     |
| OVERDUE    | DELINQUENT |
| DELINQUENT | ACTIVE     |
| ACTIVE     | CLOSED     |
| DUE        | CLOSED     |
| OVERDUE    | CLOSED     |
| DELINQUENT | CLOSED     |

## 9.3 Forbidden Transitions

Forbidden:

* PENDING → DUE
* PENDING → OVERDUE
* ACTIVE → DELINQUENT
* CLOSED → ACTIVE without explicit reopening authority
* OVERDUE → PENDING
* DELINQUENT → PENDING
* CLOSED → PENDING

No transition may occur without an auditable event.

# 11. Replay-Safe Billing Model

Billing Core must tolerate:

* duplicate generation attempts
* duplicate events
* delayed events
* out-of-order events
* partial runtime failure
* recovery replay
* reconciliation-triggered correction

## 11.1 Duplicate Invoice Handling

Duplicate invoice generation for the same subscriber, service, and billing period must resolve to one authoritative invoice.

The duplicate attempt must be logged as non-authoritative evidence.

## 11.2 Duplicate Charge Handling

Duplicate charges must be rejected unless they have distinct authorized causes.

## 11.3 Delayed Event Handling

Delayed events may be accepted only if they do not violate finalized financial truth.

If delayed events require correction, Billing Core must issue adjustment events.

## 11.4 Out-of-Order Event Handling

Billing Core must not derive invalid state from arrival order alone.

State must be derived from deterministic event authority and lifecycle rules.

# 12. Reconciliation Integration

Billing Core integrates with the Reconciliation Engine for verification, not authority surrender.

## 12.1 Reconciliation Responsibilities

Reconciliation may verify:

* invoice existence
* charge consistency
* debt consistency
* subscriber billing state
* payment application state
* eligibility-state consistency
* billing versus accounting lifecycle alignment

## 12.2 Correction Ownership

Reconciliation may detect inconsistency.

Reconciliation may not silently rewrite Billing Core.

Corrections must be executed by the authority owner.

Billing Core owns billing correction.

Payment System owns payment settlement correction.

Accounting Engine owns accounting evidence correction.

Policy Engine owns policy correction.

## 12.3 Recovery Boundaries

Billing recovery must preserve:

* invoice history
* debt history
* charge history
* credit history
* adjustment history
* eligibility transition history

Recovery must not erase financial evidence.

# 13. Operational Invariants

The following billing laws are immutable:

1. Billing Core is the only authority for financial state.
2. Billing Core must not enforce network state.
3. Billing Core must not process payments.
4. Billing Core must not mutate FreeRADIUS directly.
5. Billing Core must not mutate MikroTik directly.
6. Invoice truth must be immutable after finalization.
7. Debt must always trace back to valid invoice authority.
8. Charges must always trace back to deterministic billing cause.
9. Duplicate invoice creation is invalid.
10. Duplicate charge creation is invalid unless causally distinct.
11. Manual financial truth override is forbidden.
12. Payment evidence does not become financial truth until accepted by Billing Core.
13. Reconciliation detects inconsistency but does not own billing truth.
14. Eligibility may be determined by Billing Core but enforced only by downstream authority.
15. Closed billing state does not erase historical debt.
16. Adjustments must be append-only financial events.
17. Replay must reproduce the same billing state from the same authoritative history.
18. Billing state transition without event authority is invalid.

# 14. Forbidden Billing Patterns

The following patterns are prohibited:

* manual invoice mutation
* hidden debt creation
* duplicate invoice creation
* financial truth override
* circular billing authority
* payment system creating debt
* Radius creating billing state
* MikroTik determining billing state
* billing system enforcing network access
* direct network enforcement by Billing Core
* deleting invoices to correct mistakes
* editing paid invoices
* overwriting balances manually
* resolving debt without financial event
* treating reminder status as billing state
* treating suspension status as billing state
* treating accounting packets as invoice authority
* treating payment webhook arrival as final payment truth
* allowing UI action to bypass billing authority
* allowing admin override without auditable adjustment
